# 웹 검색 프로토콜

이 문서는 aStory 리서치를 위한 웹 검색 절차를 정의한다. 정확하고 검증 가능한 결과를 보장하기 위해 모든 검색은 이 프로토콜을 따라야 한다.

---

## 검색 전략 프레임워크

효과적인 리서치는 체계적인 검색 접근 방식을 필요로 한다. 무작위 또는 단일 쿼리 검색은 불완전한 결과를 생산한다.

### 쿼리 구성

1차 쿼리는 콘텐츠 요청자가 명시한 대로 정확한 주제를 사용한다. "Claude Code 컨텍스트 관리"와 같은 주제의 경우, 1차 쿼리는 "Claude Code 컨텍스트 관리"이다.

2차 쿼리는 범위를 확장한다. "Claude Code 토큰 제한", "Claude Code 메모리 관리", "컨텍스트 윈도우 AI 코딩 도구"가 포함될 수 있다.

3차 쿼리는 특정 세부사항을 대상으로 한다. 버전 번호, 출시 날짜, 구성 구문을 찾을 수 있다: "Claude Code 200K 토큰", "claude --context 문서".

검증 쿼리는 핵심 사실을 확증한다. 출처가 "35% 생산성 향상"을 주장하면, 검증 쿼리는 독립적인 확인을 찾는다: "Claude Code 생산성 연구", "AI 코딩 도구 벤치마크 2025".

### 쿼리 최적화

현재 연도 포함은 결과의 최신성을 향상시킨다. "2025" 또는 "2026"을 추가하면 최근 콘텐츠로 필터링된다.

도메인 제한은 결과를 집중시킨다. "site:docs.anthropic.com"을 사용하면 공식 문서를 대상으로 한다.

정확한 구문 일치는 따옴표를 사용한다. "Model Context Protocol"을 검색하면 단어를 개별적으로 검색하는 것보다 더 정확한 결과를 반환한다.

제외 연산자는 노이즈를 제거한다. 고급 콘텐츠를 찾을 때 "-tutorial" 또는 "-beginner"를 추가하면 결과가 정제된다.

## 검색 실행 프로세스

검색 프로세스는 노력을 관리하면서 범위를 최대화하기 위해 정의된 순서를 따른다.

### 1단계: 1차 검색

수정 없이 1차 쿼리를 먼저 실행한다. 상위 10개 결과를 관련성과 출처 유형에 대해 검토한다.

각 유망한 결과를 즉시 문서화한다. 진행하기 전에 URL, 제목, 명확한 콘텐츠 유형을 기록한다.

결과의 공백을 식별한다. 주제의 어떤 측면이 잘 다루어지지 않았는가? 이 공백이 2차 쿼리를 알려준다.

### 2단계: 2차 확장

1차 결과에서 식별된 공백을 기반으로 2차 쿼리를 실행한다.

주제의 다른 측면을 대상으로 한다. 1차 결과가 개념을 잘 다루었지만 예제가 부족했다면, 2차 쿼리는 예제에 집중한다.

출처 기반을 넓힌다. 1차 결과가 주로 하나의 도메인에서 왔다면, 2차 쿼리는 대안적인 권위 있는 출처를 대상으로 한다.

이미 기록된 URL을 중복하지 않고 새로운 결과를 문서화한다.

### 3단계: 특정 세부사항

콘텐츠에 필요한 사실적 세부사항을 위해 3차 쿼리를 실행한다.

특정 데이터 포인트를 대상으로 한다: 버전 번호, 통계, 날짜, 기술 사양.

주장에 대한 확증을 찾는다. 한 출처에서 인용된 통계는 원본 출처 또는 독립적인 확인에 대해 검증되어야 한다.

발견된 특정 정보를 출처 URL과 함께 문서화한다.

### 4단계: 검증

중요하거나 놀라운 주장에 대해 검증 쿼리를 실행한다.

반대 관점을 검색한다. 출처가 무언가가 유익하다고 주장하면, 비판이나 제한을 검색한다.

1차 출처를 찾는다. 블로그 포스트가 연구를 인용하면, 원본 연구를 찾는다.

각 주요 주장에 대한 검증 상태를 문서화한다: 검증됨, 부분 검증됨, 또는 미검증.

## 출처 평가 기준

모든 검색 결과가 리서치 브리프에 포함될 자격이 있는 것은 아니다. 각 잠재적 출처는 평가되어야 한다.

### 권위 평가

공식 출처가 가장 높은 권위를 받는다. 기술 벤더의 문서, 회사 엔지니어링 블로그, 공식 GitHub 저장소가 가장 권위 있다.

인정받는 전문가 출처가 높은 권위를 받는다. 확립된 기술 블로그, 컨퍼런스 발표, 알려진 전문가의 출판물이 상당한 비중을 차지한다.

커뮤니티 출처가 중간 권위를 받는다. 많은 투표를 받은 Stack Overflow 답변, 공식 응답이 있는 GitHub 이슈, 확립된 플랫폼의 튜토리얼이 유용하다.

개인 출처가 제한된 권위를 받는다. 개인 블로그 포스트는 포함 전에 추가 검증이 필요하다.

### 최신성 평가

문서는 최신이어야 한다. Claude Code와 같이 빠르게 발전하는 도구의 경우, 6개월 이상 된 출처는 오래되었을 수 있다.

버전 일치가 중요하다. 버전 1.0용 문서는 버전 2.0에 적용되지 않을 수 있다.

업데이트 날짜가 보여야 한다. 날짜가 없는 출처는 추가 주의가 필요하다.

발표 날짜는 타임라인을 확립한다. 릴리스 노트와 변경 로그는 신뢰할 수 있는 날짜 정보를 제공한다.

### 관련성 평가

주제 일치가 직접적이어야 한다. "AI 코딩 도구 일반"에 관한 출처는 Claude Code를 구체적으로 다루지 않을 수 있다.

깊이가 필요에 맞아야 한다. 더 넓은 기사에서의 짧은 언급은 충분한 세부사항을 제공하지 않을 수 있다.

대상 독자가 일치해야 한다. 관리자용 문서는 개발자 중심 콘텐츠에 적합하지 않을 수 있다.

언어 일치가 정확성을 보장한다. 기술 콘텐츠에는 영어 출처가 선호되며, 한국 특정 맥락에는 한국어 출처가 적합하다.

### 신뢰성 검증

사실적 주장은 확증이 필요하다. 주장은 여러 독립적인 출처에서 나타나야 한다.

1차 출처에 대한 링크는 신뢰성을 높인다. 자신의 출처를 인용하는 출처는 연구 엄격성을 보여준다.

오류 패턴은 신뢰성을 낮춘다. 검증할 수 있는 영역에서 명백한 사실 오류가 있는 출처는 검증할 수 없는 영역에서도 오류가 있을 수 있음을 시사한다.

상업적 편향에 주의해야 한다. 경쟁사나 벤더의 출처는 특정 측면을 강조할 수 있다.

## 반환각 프로토콜

리서치는 조작된 정보가 콘텐츠 파이프라인에 들어가는 것을 적극적으로 방지해야 한다.

### URL 검증 의무

리서치 브리프에 포함된 모든 URL은 접근 가능함이 검증되어야 한다. WebFetch 도구가 각 URL이 유효한 콘텐츠를 반환하는지 확인한다.

URL을 조작하지 않는다. URL을 찾을 수 없으면, 그럴듯한 URL을 만들어내는 대신 정보를 검증할 수 없었다고 문서화한다.

URL 구조를 가정하지 않는다. 문서 패턴이 URL이 존재해야 함을 시사하더라도, 포함 전에 검증한다.

### 콘텐츠 검증

추출된 정보는 출처 콘텐츠와 일치해야 한다. WebFetch가 페이지를 가져온 후, 특정 주장이 실제로 존재하는지 확인한다.

인용은 정확해야 한다. 출처 자료를 인용하는 경우, 의역한 후 인용으로 표시하는 대신 정확한 문구를 확인한다.

통계는 정확히 명시되어야 한다. 출처가 정확한 숫자를 명시해야 하며, 범위나 근사치를 명시한 후 지정하지 않아야 한다.

### 불확실성 공개

정보를 검증할 수 없는 경우, 추측하는 대신 불확실성을 문서화한다.

미검증 주장은 명시적으로 표시한다: "출처가 X를 주장하지만 독립적인 검증을 찾을 수 없음."

검색 제한에 주의한다: "현재 검색에서 X에 대한 결과를 찾을 수 없음; 대안적 검색 접근이 필요할 수 있음."

검증된 사실과 리서처 해석을 구분한다.

## 결과 문서화

검색 결과는 리서치 브리프를 위해 일관된 형식으로 문서화한다.

### 출처 항목 형식

각 출처 항목은 다음을 포함한다:

ID: 참조를 위한 순차 번호 (1, 2, 3).

URL: 전체 검증된 URL.

제목: 페이지 제목 또는 기사 제목.

유형: 분류 (official_docs, blog, github, video, news).

관련성: 등급 (high, medium, low).

핵심 포인트: 출처에서 추출한 정보의 불릿 목록.

검증 상태: 검증됨, 부분 검증됨, 또는 미검증.

### 핵심 포인트 추출

핵심 포인트는 각 출처에서 유용한 정보를 요약한다.

각 핵심 포인트는 별개의 정보 조각이어야 한다.

핵심 포인트는 평가적이 아닌 사실적이어야 한다.

핵심 포인트는 구체적인 세부사항을 포함해야 한다: 숫자, 날짜, 기술 용어.

핵심 포인트는 출처가 긴 경우 정보가 출처에서 어디에 나타나는지 표시해야 한다.

### 중복 제거

같은 내용을 다루는 여러 출처는 기록되어야 하지만 중복 문서화되어서는 안 된다.

2차 출처가 인용하는 경우 1차 출처가 선호된다.

같은 사실을 다루는 독립적인 출처는 확증을 제공한다.

모순되는 출처는 불일치를 기록해야 한다.

## 검색 도구 사용

WebSearch와 WebFetch 도구는 특정 패턴에 따라 사용한다.

### WebSearch 사용

초기 발견과 광범위한 탐색을 위해 WebSearch를 사용한다.

쿼리는 유용할 만큼 구체적이지만 관련 결과를 놓칠 만큼 좁지 않아야 한다.

진행하기 전에 반환된 모든 결과를 검토한다; 첫 번째 결과가 최선이라고 가정하지 않는다.

다양한 쿼리로 여러 검색을 수행하면 더 포괄적인 결과가 생산된다.

### WebFetch 사용

URL을 검증하고 상세한 콘텐츠를 추출하기 위해 WebFetch를 사용한다.

검색 후에 가져온다, 전에가 아니다. 추측된 URL에 WebFetch를 사용하면 리소스를 낭비하고 접근 불가능한 출처를 포함할 위험이 있다.

목적을 가지고 가져온다. 가져온 콘텐츠에서 답할 특정 질문이 있어야 한다.

발견한 것을 문서화한다. 추출된 정보는 리서치 브리프를 위해 즉시 기록되어야 한다.

### 도구 제한사항

검색 결과에 모든 관련 출처가 포함되지 않을 수 있다. 중요한 출처가 검색 결과에 나타나지 않을 수 있다.

가져온 페이지에 예상 정보가 포함되지 않을 수 있다. 유망한 제목이 관련 콘텐츠를 전달하지 않을 수 있다.

동적 콘텐츠는 캡처되지 않을 수 있다. 일부 페이지는 콘텐츠를 표시하기 위해 상호작용이 필요하다.

유료 콘텐츠는 접근 불가능할 수 있다. 접근 제한을 존중한다.

## 품질 보증

검색 품질은 리서치 브리프를 편집하기 전에 검증한다.

### 범위 검사

주제의 모든 주요 측면이 출처 지원을 받아야 한다.

계획된 콘텐츠에서 중요한 주장이 출처 뒷받침 없이 있어서는 안 된다.

대안적 관점이 존재하는 경우 대표되어야 한다.

### 검증 검사

모든 높은 관련성 출처는 URL 검증이 되어야 한다.

모든 통계와 특정 주장은 검증이 기록되어야 한다.

미검증 주장은 명시적으로 표시되어야 한다.

### 편향 검사

출처 유형이 다양해야 하며, 하나의 도메인이나 관점에서만 오지 않아야 한다.

상업적 이해관계가 관련되는 경우 기록되어야 한다.

반대 의견이 인정되어야 한다.

### 최신성 검사

출처가 주제에 충분히 최신이어야 한다.

오래된 정보는 그렇게 표시되어야 한다.

버전 특정 정보는 버전을 기록해야 한다.
