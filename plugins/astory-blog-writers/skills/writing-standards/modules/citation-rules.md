# 인용 시스템 규칙

이 문서는 aStory 블로그 콘텐츠를 위한 인용 시스템을 정의한다. 모든 사실적 주장은 신뢰성을 유지하고 환각을 방지하기 위해 적절히 출처를 표기해야 한다.

---

## 인용 번호 시스템

인용 시스템은 본문 내에서 번호가 매겨진 참조를 사용하며, 각 챕터 끝에 완전한 참고문헌 섹션을 포함한다.

### 본문 내 인용 형식

인용은 참조하는 콘텐츠 바로 뒤에 괄호 안의 숫자로 나타난다. 형식은 (1), (2), (3) 등이다.

단일 인용은 마침표 앞 문장 끝에 번호를 배치한다: "Claude Code 지원 터미널 기반의 AI 코딩 도구다 (1)."

다른 출처의 복수 인용은 쉼표로 구분한다: "이 기능은 생산성을 35% 향상시킨다 (1, 2, 3)."

같은 출처가 문단 내 여러 문장을 지원할 때 인용은 지원되는 콘텐츠의 마지막 문장 뒤에 나타난다.

### 섹션 기반 번호 매기기

인용 번호는 각 섹션 내에서 순차적이다. 인용된 첫 번째 출처는 (1), 두 번째는 (2)이며, 챕터 위치와 관계없다.

같은 출처가 섹션 내에서 여러 번 참조될 때 매번 같은 번호를 사용한다. 작가는 어떤 출처에 어떤 번호가 할당되었는지 추적한다.

새로운 섹션은 (1)부터 번호 매기기를 다시 시작한다. 이는 긴 챕터에서 번호가 다루기 어려워지는 것을 방지한다.

### 배치 가이드라인

인용은 자연스러운 문장 경계에 나타난다. 절의 흐름을 중단하지 않는다.

기술 사양의 경우 인용은 사양 뒤에 온다: "200K 토큰 컨텍스트 창 (1)."

통계적 주장의 경우 인용은 통계 뒤에 온다: "생산성이 78% 증가되었다 (2)."

프로세스 설명의 경우 전체 문단이 하나의 출처에서 콘텐츠를 설명한다면 인용은 문단 뒤에 올 수 있다.

## 참고문헌 섹션 형식

각 챕터는 해당 챕터에서 인용된 모든 출처를 나열하는 "참고 문헌" 섹션으로 끝난다.

### 섹션 헤더

참고문헌 섹션은 2단계 헤딩을 사용한다: "## 참고 문헌".

이 섹션은 챕터 요약 뒤, "다음 단계" 섹션 앞에 나타난다.

### 항목 형식

각 참고문헌 항목은 다음 구조를 따른다: "[번호]. [제목] - [URL]"

예시 항목:

1. Claude Code 개요 - https://docs.anthropic.com/ko/docs/claude-code/overview
2. Claude Code 기능 소개 - https://docs.anthropic.com/ko/docs/claude-code/features
3. 에이전틱 코딩 모범 사례 - https://www.anthropic.com/engineering/claude-code-best-practices

### 항목 구성 요소

번호: 본문 내 인용 번호와 정확히 일치한다.

제목: 로컬 문서 파일의 문서 제목 또는 설명적 섹션 제목을 사용한다.

URL: 로컬 문서 헤더에서 추출한 출처 URL.

### 순서

참고문헌은 본문에서 첫 인용의 순서와 일치하는 번호순으로 나타난다.

본문에서 사용된 모든 번호는 해당하는 항목이 있어야 한다. 본문 내 인용 없이 항목이 존재해서는 안 된다.

## 로컬 문서 시스템

aStory는 정확성을 보장하고 오프라인 작업을 가능하게 하기 위해 공식 문서의 로컬 복사본을 사용한다.

### 문서 위치

로컬 문서는 다음 위치에 있다: .claude/docs/claude-code-docs/

컬렉션에는 참조용으로 다운로드하고 유지 관리하는 33개의 공식 문서가 포함되어 있다.

### 출처 URL 추출

각 로컬 문서는 출처 URL이 포함된 헤더를 포함한다:

```
# 문서 제목

> **출처**: https://docs.anthropic.com/ko/docs/claude-code/overview
> **다운로드 일자**: 2025-12-03
```

작가는 참고문헌 섹션을 위해 "출처" 줄에서 URL을 추출한다.

다운로드 날짜는 인용의 접근 날짜로 사용된다.

### 문서 카테고리

기본 문서 (01-05): 개요, 빠른 시작, 일반 워크플로우, 파일과 이미지, 메모리와 컨텍스트.

인터페이스 문서 (06-10): CLI 사용법, IDE 통합, SDK, 멀티 모델 지원, 커스텀 명령.

에이전트 문서 (11-15): 도구와 권한, 확장 사고, MCP, 서브에이전트, 에이전틱 루프.

자동화 문서 (16-20): GitHub Actions, 헤드리스 모드, 훅, 컨텍스트 관리, 모델 설정.

고급 문서 (21-25): 에이전트 상세, 규칙 시스템, 설정, 비용, 개인정보와 보안.

참조 문서 (26-30): 트러블슈팅, API 참조, 변경 로그, FAQ, 용어집.

모범 사례 문서 (31-33): 프롬프팅 모범 사례, 에이전틱 코딩 모범 사례, 메모리 관리 모범 사례.

## 필수 vs 선택 인용

모든 진술이 인용을 필요로 하는 것은 아니다. 이 섹션은 인용이 필수인 경우를 명확히 한다.

### 필수 인용

통계와 수치 데이터는 인용이 필요하다: "78% 개발자가..."

공식 정의와 용어는 Claude 특정 개념을 소개할 때 인용이 필요하다.

기술 사양은 인용이 필요하다: "200K 토큰", "v1.0.58".

릴리스 날짜와 버전 정보는 인용이 필요하다.

공식 명령과 코드 구문은 소개할 때 인용이 필요하다.

벤치마크 결과는 인용이 필요하다.

도구 비교는 사실적 차이를 제시할 때 인용이 필요하다.

공식 발표와 회사 성명은 인용이 필요하다.

### 인용 불필요

일반 상식은 인용이 필요하지 않다: "코딩은 프로그래밍 언어로..."

작가 분석과 해석은 인용이 필요하지 않다. 작가가 인용된 사실에서 결론을 도출할 때 그 결론은 작가의 기여이다.

작가가 만든 비유와 예시는 인용이 필요하지 않다: "마치 자율 주행 자동차처럼..."

인용된 사실에서의 논리적 추론은 인용이 필요하지 않다: "따라서", "그러므로".

### 경계 사례

업계 표준 관행은 그 관행이 얼마나 보편적인지에 따라 인용이 필요하거나 필요하지 않을 수 있다.

널리 문서화된 역사적 사실은 인용이 필요하지 않을 수 있지만, 인용을 추가하면 신뢰성이 강화된다.

불확실한 경우 작가는 인용을 추가하는 쪽으로 해야 한다.

## 인용 워크플로우

작가는 콘텐츠 작성 전반에 걸쳐 적절한 인용을 보장하기 위해 이 프로세스를 따른다.

### 1단계: 사전 작성 리서치

작성을 시작하기 전에 작가는 챕터 콘텐츠와 관련된 로컬 문서를 식별한다.

챕터 계획은 참조 문서를 지정한다. 작가는 이 문서들이 존재하고 필요한 정보를 포함하는지 확인한다.

작가는 나중에 사용하기 위해 각 문서 헤더의 출처 URL을 기록한다.

### 2단계: 작성 중 인용

콘텐츠를 구성하면서 작가는 즉시 인용 지점을 표시한다.

출처에서 사실을 진술할 때 작가는 나중에 돌아오는 대신 바로 인용 번호를 삽입한다.

작가는 출처와 할당된 번호의 진행 목록을 유지한다.

### 3단계: 참고문헌 섹션 편집

본문 텍스트를 완료한 후 작가는 참고문헌 섹션을 편집한다.

본문의 각 인용 번호는 일치하는 항목이 있어야 한다.

작가는 URL이 로컬 문서 헤더에서 정확히 복사되었는지 확인한다.

### 4단계: 검증

작가는 모든 본문 내 번호가 참고문헌 섹션 항목과 일치하는지 검토한다.

작가는 본문에 나타나지 않고 참고문헌에 인용된 출처가 없는지 확인한다.

작가는 출처 URL이 접근 가능하고 올바른 콘텐츠를 가리키는지 확인한다.

## 일반적인 인용 오류

작가는 인용 프로세스에서 이러한 빈번한 실수를 피한다.

### 번호 불일치

오류: 본문에서 (3)을 사용하지만 참고문헌 섹션에 항목 3이 없음.

예방: 작성 중 진행 목록을 유지하고 완료 전에 조정한다.

### 사실에 대한 인용 누락

오류: "생산성이 35% 향상된다"를 인용 없이 진술.

예방: 모든 통계에 대해 "이 숫자는 어디서 왔는가?"라고 질문한다.

### 과잉 인용

오류: 일반 상식이나 명백한 진술에 인용.

예방: "출처 없이 이것이 의문시될 것인가?" 테스트를 적용한다.

### 깨진 URL

오류: 더 이상 작동하지 않는 URL 포함.

예방: 최종 검토 과정에서 모든 URL을 확인한다.

### 일관성 없는 형식

오류: 인용 형식 혼합: (1), [1], 1.

예방: 전체에서 괄호 형식 (1)만 사용한다.

## 검증 체크리스트

제출 전 작가는 모든 인용 요구사항을 확인한다:

제출 전 검사 포함:

모든 통계에 인용 번호가 있다.

모든 기술 사양에 인용 번호가 있다.

모든 공식 용어는 첫 사용 시 인용 번호가 있다.

참고문헌 섹션이 챕터 끝에 존재한다.

모든 본문 내 번호는 일치하는 참고문헌 항목이 있다.

모든 참고문헌 항목은 일치하는 본문 내 인용이 있다.

출처 URL은 로컬 문서에서 추출되었다.

URL은 접근 가능한 것으로 확인되었다.

조작되거나 가정된 URL이 나타나지 않는다.

검증되지 않은 환각 사실이 나타나지 않는다.
