# 전문 영역: 아키텍처 (Architecture Expertise)

## 개요

아키텍처 전문 영역은 시스템 설계, 아키텍처 패턴, 분산 시스템, 확장성 전략, 기술적 의사결정 프레임워크를 포괄한다. 이 영역은 소프트웨어 아키텍처에 대한 권위 있는 콘텐츠 작성을 위한 지식 기반을 제공한다.

## 핵심 지식 영역

- 계층형 아키텍처 패턴 (프레젠테이션, 비즈니스, 데이터)
- 서비스 지향 아키텍처 (SOA)
- 마이크로서비스 아키텍처
- 이벤트 기반 아키텍처
- 헥사고날 아키텍처 (포트와 어댑터)
- 클린 아키텍처와 의존성 역전

아키텍처 스타일:

## 전문 어휘

아키텍처 용어:
- 컴포넌트 (component) - 모듈식 구성 요소
- 계층 (layer) - 아키텍처 티어
- 경계 (boundary) - 시스템 또는 컨텍스트 경계
- 결합도 (coupling) - 상호 의존 정도
- 응집도 (cohesion) - 모듈 내 관련성 정도
- 인터페이스 (interface) - 컴포넌트 간 계약

패턴 용어:
- 패턴 (pattern) - 재사용 가능한 솔루션 템플릿
- 전략 (strategy) - 접근 방식 또는 방법론
- 아키텍처 스타일 (architectural style) - 상위 수준 구조
- 설계 원칙 (design principle) - 지침 규칙
- 트레이드오프 (tradeoff) - 균형 잡힌 절충

분산 시스템 용어:
- 일관성 (consistency) - 노드 간 데이터 일치
- 가용성 (availability) - 시스템 가동 시간과 접근성
- 분할 내성 (partition tolerance) - 네트워크 분리에 대한 회복력
- 복제 (replication) - 신뢰성을 위한 데이터 복사
- 샤딩 (sharding) - 확장성을 위한 데이터 분할

## 개념적 프레임워크

SOLID 원칙:
- 단일 책임 원칙 (Single Responsibility Principle)
- 개방-폐쇄 원칙 (Open/Closed Principle)
- 리스코프 치환 원칙 (Liskov Substitution Principle)
- 인터페이스 분리 원칙 (Interface Segregation Principle)
- 의존성 역전 원칙 (Dependency Inversion Principle)

도메인 주도 설계 (DDD):
- 바운디드 컨텍스트 (Bounded contexts)
- 애그리거트와 엔티티 (Aggregates and entities)
- 값 객체 (Value objects)
- 도메인 이벤트 (Domain events)
- 유비쿼터스 언어 (Ubiquitous language)

클라우드 네이티브 원칙:
- 12요소 앱 방법론
- 컨테이너 기반 배포
- 선언적 구성
- 상태 없는 프로세스
- 일회성과 빠른 시작

아키텍처 심층 분석 구조:

이론적 기초:

패턴 분석:
- 아키텍처 패턴을 체계적으로 제시
- 컴포넌트와 상호작용 분석
- 구현 변형 논의

트레이드오프 평가:
- 대안을 객관적으로 비교
- 성능 영향 분석
- 운영 복잡도 고려

구현 지침:
- 실용적 권고사항 제공
- 일반적인 함정 해결
- 점진적 도입 경로 제안

결론:
- 핵심 통찰 종합
- 명확한 권고사항 제시
- 향후 고려사항 식별

## 일반적인 논증 접근법

원칙 기반 추론:
"이 설계는 단일 책임 원칙을 따른다. 각 컴포넌트가 하나의 변경 이유만 가지므로, 시스템의 유지보수성이 향상된다."

트레이드오프 분석:
"마이크로서비스는 독립적 배포와 확장성을 제공하지만, 분산 시스템 복잡도와 운영 오버헤드를 증가시킨다. 따라서 팀 규모와 도메인 복잡도를 고려한 의사결정이 필요하다."

패턴 적용:
"Saga 패턴은 분산 트랜잭션 문제에 대한 실용적 해법이다. 보상 트랜잭션을 통해 ACID 대신 최종 일관성을 확보한다."

- 운영 관련 사항을 다룬다
